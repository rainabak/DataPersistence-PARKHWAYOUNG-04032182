# PRD — DataPersistence PoC

## 개요

MVC Skeleton에 JSON 파일 기반 Persistence Layer를 추가하여 DB 없이 데이터가 유지되는지 검증한다.

---

## 배경

현재 프로젝트는 MVC 구조가 완성된 상태다. Controller, View, Model 계층이 분리되어 있고, `ItemRepository` 인터페이스가 정의되어 있다. 그러나 실제 저장소 구현이 없어 데이터가 메모리에만 유지되고 재실행 시 사라진다.

이번 PoC의 목표는 저장 방식을 검증하는 것이 아니라, **Persistence 계층을 MVC에 올바르게 통합하는 구조**를 검증하는 것이다.

---

## PoC 목표

| 번호 | 목표 | 성공 기준 |
|---|---|---|
| 1 | Persistence 계층 분리 | Controller가 저장 방식을 모른다 |
| 2 | Repository 패턴 검증 | 인터페이스 교체 없이 구현체만 바꿀 수 있다 |
| 3 | CRUD 구조 검증 | 등록, 조회, 수정, 삭제가 모두 동작한다 |
| 4 | 재실행 후 데이터 유지 | 앱 재시작 후 이전 데이터가 그대로 보인다 |
| 5 | 파일 안전성 | 비정상 종료 후에도 파일이 손상되지 않는다 |

---

## 제약사항

- DB 사용 금지 (SQLite, PostgreSQL, 외부 DB 서버 포함)
- JSON 파일 기반으로만 구현
- 외부 라이브러리 없이 표준 C++17만 사용
- 외부 라이브러리가 필요한 경우 이유를 먼저 설명하고 승인받을 것

## 데이터 모델

### Item

| 필드 | 타입 | 설명 |
|---|---|---|
| id | 정수 | 자동 증가, 재실행 후에도 연번 유지 |
| name | 문자열 | 항목 이름 |
| price | 실수 | 항목 가격 |

---

## Persistence 설계 요구사항

### JSON 파일 저장 방식

- 저장 위치: 실행 파일 기준 `data/items.json`
- 저장 단위: 전체 컬렉션을 하나의 파일에 저장
- 저장 시점: CRUD 연산 직후 즉시 파일에 반영
- `nextId` 를 파일에 함께 저장하여 재실행 후에도 ID 연번을 유지한다

### 쓰기 전략

- 전체 덮어쓰기 방식(Full Overwrite)을 사용한다
- 임시 파일(`.tmp`)에 먼저 쓴 후 원본 파일로 교체하여 쓰기 중 손상을 방지한다

### 읽기 전략

- 앱 시작 시 한 번 파일을 읽어 메모리 벡터에 적재한다
- 이후 조회는 메모리에서 처리하고 파일을 재읽지 않는다

---

## 예외 처리 요구사항

| 상황 | 처리 |
|---|---|
| `data/` 폴더 없음 | 자동 생성 |
| `items.json` 없음 | 정상 초기 상태로 간주, 빈 데이터로 시작 |
| JSON 파일 손상 | 빈 데이터로 시작, 손상 파일을 `.bak`으로 보관, 경고 출력 |
| ID 미존재 (update/delete) | 실패 반환, 사용자에게 메시지 출력 |
| 파일 쓰기 실패 | 예외 발생, 사용자에게 저장 실패 알림 |

---

## 계층 구조 요구사항

```
Controller
  → IItemRepository (인터페이스)
      → JsonItemRepository (구현체)
          → JsonFileStore (파일 I/O)
              → data/items.json
```

- Controller는 `IItemRepository`만 의존한다
- `JsonItemRepository`는 CRUD 로직을 담당한다
- `JsonFileStore`는 파일 읽기/쓰기와 JSON 직렬화만 담당한다

---

## 비기능 요구사항

- 파일 I/O 실패는 사용자에게 명확한 메시지로 전달한다
- 정상 종료, 비정상 종료 모두에서 마지막으로 성공한 쓰기 시점의 데이터가 보존된다
- 코드 변경 없이 JSON → 다른 저장소로 교체 가능한 구조를 유지한다
